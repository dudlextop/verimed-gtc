# План реализации редизайна Verimed

Статус: выполнено. Все девять фаз завершены и прошли финальную QA.

Дата: 16 июля 2026 года.

## 1. Цель и неизменяемые ограничения

План приводит текущий frontend к утверждённым референсам без изменения:

- аналитики, формул и результатов;
- маршрутов и структуры основной навигации;
- пользовательских сценариев;
- русскоязычной терминологии;
- server pagination, фильтрации и рабочего контекста;
- логики экспертных решений и графа связей.

Разрешены небольшие аддитивные изменения backend/API для регионального агрегата и CSV-экспорта. Они обязаны переиспользовать существующие snapshots, фильтры и сортировку; новые аналитические формулы, модели данных и миграции не требуются.

Функции из PNG не копируются автоматически. Каждая проходит продуктовую оценку и получает одно из решений: реализовать, реализовать ограниченно, отложить или отклонить.

## 2. Стратегия

Редизайн выполняется снизу вверх:

1. зафиксировать текущее поведение и визуальные baseline;
2. внедрить токены и бренд-актив;
3. обновить shell и примитивы;
4. переработать главный рабочий сценарий проверки;
5. перенести остальные object/list screens;
6. завершить обзорные и поддерживающие страницы;
7. провести сквозную responsive, accessibility и visual QA.

Такой порядок не допускает расхождения нескольких параллельных визуальных систем и снижает риск затронуть бизнес-логику.

## 2.2. Проверка каждого нового элемента

До реализации владелец изменения отвечает в PR:

1. Какой пользовательский вопрос решается?
2. Почему существующего элемента недостаточно?
3. Откуда поступают данные?
4. Где элемент находится в сценарии?
5. Можно ли скрыть его на втором уровне или загрузить лениво?
6. Не дублирует ли он другой блок?

Отсутствие проверяемого ответа блокирует включение элемента.

## 2.3. Необходимые изменения данных и API

### Аддитивное расширение overview

`GET /api/analytics/overview` получает:

- `timeline` из уже существующего сервиса;
- `regional_monitoring[]`: каноническое название региона, число сигналов, число уникальных связанных записей, финансовая значимость, число организаций, максимальный приоритет и ведущая организация;
- `schema_version` повышается при изменении ответа.

Значения относятся к последнему завершённому запуску и собираются из готовых аналитических снимков и текущих сущностей. Количество сигналов и уникальных записей остаются разными полями. Для отсутствующего региона сервер не создаёт нулевую «статистику»: frontend сопоставляет геометрию с массивом и показывает «Нет данных».

### Экспорт

Добавляются аддитивные export endpoints:

- `GET /api/exports/signals.csv` — текущая выборка по тем же query params;
- `POST /api/exports/signals.csv` — выборка по явному списку `signal_ids`;
- `GET /api/exports/organizations.csv` — текущая выборка по тем же query params.

Они:

- принимают те же filters и sort, что и списки;
- повторно используют общий SQL/query builder;
- не ограничиваются текущей страницей;
- не принимают `page`/`page_size` как ограничение экспортируемой выборки;
- для выбранных сигналов принимают явный список IDs;
- отдают UTF-8 CSV с русскими заголовками и безопасным filename;
- имеют верхний limit и защиту от CSV formula injection;
- не возвращают технические fingerprints, hash и лишние персональные сведения.

Выбор сигналов сохраняется при смене страницы внутри неизменной filter signature. При изменении filters выбор очищается с коротким уведомлением, чтобы скрытые строки не попали в файл неожиданно.

### Профиль

Backend не изменяется. Локальная схема versioned, проверяется frontend-валидаторами и может быть сброшена. Ограничение хранения показывается в форме и документации.

### Модели данных

Новые таблицы и миграции не нужны. Геометрия карты хранится как статический frontend asset и не содержит аналитических значений.

- перед графом вывести вывод и основных участников;
- входящие сигналы открывают существующий quick preview;
- закреплённая зона решения использует один primary action;
- предыдущая/следующая модель сохраняют текущий список.

### Граф связей

- не менять graph data, layout algorithm и selection state;
- переназначить node palette без purple;
- усилить центральный объект;
- ослаблять нерелевантные связи при выборе;
- обновить legend и tooltip;
- сохранить «Показать больше связей»;
- оставить mobile list главным представлением.

### Тесты и приёмка

- graph nodes/edges те же до и после;
- выбор узла не выполняет лишнюю навигацию;
- mobile list доступен без canvas;
- quick preview возвращается в модель;
- snapshots списка и пяти состояний detail screen.

## 10. Фаза 6. Сводная аналитика и обзор

## 16. Риски и способы снижения

| Риск | Снижение |
|---|---|
| Одновременный большой CSS diff | переносить по primitives и routes, удалять старые стили после миграции |
| Конфликт новых tokens со старыми | временный migration map и запрет новых legacy classes |
| Визуальный refactor меняет обработчики | behavioral tests до и после каждого PR |
| Logo PNG теряет качество | запросить SVG или использовать retina crop без перерисовки |
| Функции из референса раздувают scope | продуктовый шлюз, утверждённая матрица и запрет добавлять элементы вне неё |
| CSV расходится с таблицей | общий query builder, contract tests на одинаковых filters/sort |
| Региональная карта показывает неверное соответствие | canonical region mapping, тест покрытия геометрии и явное «Нет данных» |
| Локальный профиль воспринимается как учётная запись | постоянное пояснение хранения, отсутствие прав/пароля и versioned reset |
| Графики перегружают страницы | максимум два ведущих графика, lazy secondary blocks и вопрос в заголовке |
| Карточки становятся слишком плотными | проверка длинных данных и 200% zoom |
| Graph restyle меняет topology | snapshot данных узлов/рёбер и отдельный visual renderer test |

## 17. Принятые решения и итог

7. Используется оптимизированный retina PNG с сохранением исходного artwork.
