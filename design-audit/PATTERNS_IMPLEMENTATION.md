# Реализация Verimed V2: повторяющиеся модели

## Границы фазы

Фаза `design-v2-patterns` мигрирует только `/patterns`, `/patterns/[id]` и общие presentation-компоненты, необходимые этому сценарию. Backend, API-контракты, аналитические формулы, база данных, миграции, маршруты и topology графа не изменены.

Все значения продолжают поступать из существующих API:

- `/patterns`;
- `/analytics/pattern-summary`;
- `/analytics/pattern-changes`;
- `/patterns/{id}`;
- `/patterns/{id}/timeline`;
- `/patterns/{id}/recurrence-history`;
- `/patterns/{id}/graph`;
- `/patterns/{id}/signals`;
- существующих decision и review endpoints.

## Список `/patterns`

Страница использует `PageHeader`, `MetricStrip`, `InlineNotice`, `FilterBar`, `DataTableShell`, `MobileObjectCard`, `DomainIndicator`, `FinancialValue`, `OverflowActions`, `EmptyState` и list-skeleton.

- три KPI собраны в одну компактную полосу;
- фильтры важности, устойчивости, типа и организации находятся на первом уровне;
- статус, период, финансовый диапазон и сортировка раскрываются во втором уровне;
- URL остаётся единственным источником filter, sort и pagination state;
- поиск не добавлен, поскольку текущий backend его для моделей не поддерживает;
- строка целиком открывает модель и передаёт `returnTo` и локальную очередь идентификаторов;
- до 768 px таблица заменяется `MobileObjectCard`;
- важность, устойчивость, финансовая значимость и экспертный статус не оформляются одинаковыми бейджами.

## Карточка `/patterns/[id]`

Длинная карточка разделена на пять смысловых секций:

1. Сводка.
2. Почему модель важна.
3. Участники и связи.
4. Граф связей.
5. Входящие сигналы и экспертное решение.

Локальный selector использует якоря без отдельного tab-state, обозначает активный раздел через `aria-current="location"` и прокручивается горизонтально на mobile.

### Сводка

Важность является ведущим показателем. Устойчивость, финансовая значимость и период показаны как отдельные вторичные домены. Ниже выводятся только три факта: число сигналов, организаций и основных участников.

### Факторы и динамика

По умолчанию отображаются три ведущих фактора. Остальные раскрываются кнопкой с `aria-expanded`. Каждый фактор использует единый порядок: группа, название, вклад, наблюдаемое значение, сопоставимое значение и объяснение.

Динамика использует существующую recurrence history для фактических значений важности и устойчивости. Пропущенные точки не достраиваются. Существующий timeline API даёт дополнительный контекст по числу сигналов и финансовой значимости периода. Если появление модели зафиксировано менее двух раз, отображается честное состояние недостаточности данных.

### Участники

Участники показаны до графа и сгруппированы по фактическим категориям API:

- медицинские организации;
- врачи;
- услуги;
- обезличенные пациенты.

На первом уровне выводятся четыре элемента каждой категории. Остальные доступны через disclosure. Для врачей и пациентов сохраняются технические обезличенные коды; персональные имена не добавляются.

## Relationship graph

Алгоритм размещения, состав узлов, рёбер, идентификаторы, selection state и правило раскрытия сохранены без изменений. Переработан только visual layer:

- центральная модель — насыщенный синий;
- организация — светлый синий;
- врач — cyan;
- пациент — серо-синий;
- услуга — teal;
- сигнал — янтарный;
- рёбра — холодный серо-голубой.

При выборе узла усиливаются сам узел и ближайшие связи, остальные элементы ослабляются. Узлы и рёбра получили диагностические data-атрибуты для проверки неизменности topology.

На mobile текстовый список связей является первым представлением. Canvas создаётся только после явного раскрытия «Открыть визуальный граф», поэтому маленький интерактивный холст не блокирует понимание структуры и не расходует ресурсы без действия пользователя.

Контекст выбранного узла служит клавиатурной альтернативой tooltip и показывает тип, подпись, число сигналов, финансовую значимость и рабочий переход при его наличии. Неизвестный тип узла получает нейтральный fallback и не окрашивается случайной семантикой.

## Входящие сигналы и quick preview

Список сигналов переиспользует review-flow patterns:

- `DataTableShell` на desktop;
- `MobileObjectCard` на mobile;
- `DomainIndicator` для приоритета и статуса;
- `FinancialValue` для суммы;
- существующий `SignalPreviewPanel`.

Строка или карточка открывает быстрый просмотр. Previous/next используют полную фактическую выборку сигналов модели, даже если на первом уровне показаны только первые восемь записей. Полная карточка сигнала получает контекст возврата в модель.

## Решение и StickyActionBar

Экспертная форма, причины, feedback, decision history, recurrence history и типы решений сохранены. Диалог переведён на V2, получил focus trap, Escape, блокировку фоновой прокрутки и возврат фокуса.

`StickyActionBar` содержит одну primary CTA. После сохранения решения primary CTA меняется на переход к следующей модели, если она существует в текущей очереди. Остальные рабочие действия находятся в `OverflowActions`.

Общий `DecisionTimeline` переведён с violet-оформления на V2 blue/cyan-токены. Это минимальный совместимый shared-component fix; его данные и поведение не изменены.

## Accessibility и responsive

Проверяются:

- keyboard opening строк, карточек и узлов;
- доступные фильтры и server sorting;
- `aria-current` локальной навигации;
- `aria-expanded` disclosures;
- `aria-pressed` выбранного узла;
- текстовый mobile fallback графа;
- Escape, focus trap и return focus quick preview и review dialog;
- touch targets не меньше 44 px;
- `prefers-reduced-motion`;
- zoom 200%;
- отсутствие document overflow на 1440, 1280, 1024, 768 и 375 px.

Финальный frontend-набор содержит 139 успешно пройденных тестов в 30 файлах. Специализированная visual-матрица создала 43 снимка; после tablet-исправления дополнительно созданы целевые снимки списка на 1024 и 1280 px. Headless Chrome не обнаружил горизонтального переполнения документа, необработанных исключений или ошибок браузерной консоли.

## Legacy

Из patterns flow удалены:

- тёмная шапка таблиц;
- violet hover и violet/indigo оформление графа;
- radial glow графа;
- локальные тяжёлые KPI-карточки;
- legacy sticky workbar;
- monospace для обычных аналитических чисел;
- отдельная несовместимая таблица входящих сигналов.

Legacy styles остальных маршрутов сохранены. Backend не затронут.
