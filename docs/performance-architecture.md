# Архитектура производительности

## Контур чтения

Публичный backend обслуживает подготовленный snapshot и не импортирует аналитический
конвейер для GET-запросов. При холодном старте выполняются только проверка bundled-файла,
однократное защищённое копирование в writable-каталог и создание SQLAlchemy engine.
Генерация, миграции, Isolation Forest, расчёт риска и построение моделей доступны только
через явные служебные команды.

Зависимости `pandas` и `scikit-learn` вынесены в optional extra `analysis`. Локальная
разработка устанавливает `.[dev,analysis]`, Docker — `.[analysis]`, а Vercel serving
service не включает ML-пакеты в Python function bundle. Поэтому локальная команда полного
анализа сохранена, но публичный showcase предназначен для чтения заранее рассчитанного
snapshot, а не для повторного обучения внутри serverless function.

## Агрегаты страниц

- `/api/analytics/home` — данные первого и второго уровня сводной аналитики;
- `/api/analytics/overview` — управленческий обзор, top-N и распределение моделей;
- `schema_version` позволяет изменять состав presentation-ответа без скрытого изменения
  аналитической методики.

Агрегаты вызывают существующие сервисы и не содержат отдельной формулы. Результат
коалесцируется под `RLock` и хранится 30 секунд внутри instance. После POST-решения или
явного запуска анализа cache полностью инвалидируется.

## Пагинация и payload

`page` начинается с 1, `page_size` ограничен API (по умолчанию 20, максимум 100).
Фильтрация, сортировка, `offset` и `limit` применяются в SQL. List DTO содержит только
данные строки: идентификатор, организацию, услугу, дату, основной вывод, статус, риск,
приоритет и финансовую значимость. Факторы, связанные записи, граф и журнал загружаются
отдельно.

## Индексы

Миграция `0007` добавляет индексы:

- `risk_signals(status, level, created_at)`;
- `review_priorities(analysis_run_id, score)`;
- `review_priorities(level, score)`;
- `review_priorities(financial_significance)`;
- `organization_priority_snapshots(analysis_run_id, score, organization_id)`;
- `recurring_patterns(is_active, importance_score, id)`;
- `recurring_patterns(is_active, stability_score, id)`.

`EXPLAIN QUERY PLAN` подтверждает использование индексов для сортировки сигналов по
приоритету и моделей по важности. Сортировка 20 организаций может использовать временный
B-tree, но выполняется на малом фиксированном наборе и не оправдывает дополнительный
индекс или денормализацию.

## HTTP и клиентский cache

Общие неизменяемые аналитические и методические ответы получают
`public, max-age=60, s-maxage=300, stale-while-revalidate=86400`. Ответы с текущим
экспертным статусом, журнал, health и все POST имеют `no-store`. Ошибки не кэшируются.

Frontend использует единый `fetch`-клиент:

- объединяет одновременно выполняемые одинаковые GET;
- хранит аналитические справочные ответы до пяти минут, list/detail — до 30 секунд;
- отменяет предыдущий list-запрос той же группы при смене фильтра;
- очищает cache после POST;
- не выполняет prefetch для сотен строк таблиц.

## Progressive loading

Заголовок и основной объект карточки появляются после одного detail-запроса. История,
повторные появления, сравнение и модели организации не блокируют основной результат.
Recharts и граф связей подключаются через dynamic import; граф, timeline и связанные
сигналы модели запрашиваются только при приближении раздела к viewport. Поиск журнала
имеет debounce 350 мс.

## Пересоздание snapshot

После осознанного изменения схемы или синтетического набора:

```bash
cd backend
python scripts/build_showcase_snapshot.py
DATABASE_URL=sqlite:////absolute/path/to/backend/app/showcase/verimed-showcase.db alembic upgrade head
```

Затем необходимо обновить `SHOWCASE_STORAGE_VERSION`, проверить `alembic_version`, размер
файла, основные GET и неизменность контрольных метрик. Runtime никогда не изменяет
bundled snapshot.

## Наблюдаемые ограничения Vercel

- cache в памяти не общий между Python instances;
- временная SQLite-копия и экспертные события могут исчезнуть после recycle;
- cold start остаётся платформенным ограничением, хотя serving import отделён от ML;
- для постоянной эксплуатации необходима внешняя база и отдельный lifecycle миграций.
