# Архитектура Verimed

## Контуры приложения

Verimed — монорепозиторий из Next.js и FastAPI. PostgreSQL используется при запуске через Docker Compose, SQLite — в тестах и локальной валидации.

```text
Синтетический каталог
  → генератор нормальных записей
  → контролируемые внедрения аномалий и эталонная разметка
  → детерминированные проверки
  → признаки организаций и сопоставимые группы
  → Isolation Forest
  → формула риска и объяснения
  → метрики качества
  → FastAPI
  → Next.js
```

API-маршруты выполняют валидацию и делегируют работу сервисам. Бизнес-логика находится в `backend/app/services`, генерация — в `backend/app/synthetic`.

## Слои анализа

- `synthetic/generator.py` создаёт только нормальные записи;
- `synthetic/injectors.py` содержит восемь независимых функций внедрения;
- `services/detection.py` выполняет детерминированные проверки;
- `services/organization_model.py` формирует признаки, сопоставимые группы и оценки Isolation Forest;
- `services/risk.py` рассчитывает итоговую оценку и вклады факторов;
- `services/financial_priority.py` рассчитывает финансовую значимость и приоритет проверки;
- `services/recurring_patterns.py` группирует сигналы, связывает версии по fingerprint и строит граф;
- `services/fingerprints.py` канонизирует устойчивую идентичность сигнала;
- `services/expert_decisions.py` добавляет события, восстанавливает статус, строит журнал и проверяет hash-цепочку;
- `services/analysis_pipeline.py` оркестрирует запуск и сохраняет метрики.

## Хранение

`MedicalRecord` содержит внутреннюю эталонную метку. `GroundTruthAnomaly` фиксирует параметры внедрения и связанные записи. `RiskSignal` хранит аналитический результат и устойчивый fingerprint. Экспертное решение вынесено в независимый append-only `ExpertDecisionEvent`.

Признаки и модельные оценки сохраняются в `OrganizationFeature` и `OrganizationAnomalyScore`. Метрики каждой версии анализа хранятся в `AnalysisMetric` и связаны с `AnalysisRun`.

### Снимки запусков анализа

`AnalysisRun` также хранит идентификаторы отобранных записей, идентификаторы записей высокого риска, оценки организаций, сумму услуг к проверке, долю отбора и число завершённых проверок. Эти снимки позволяют сравнить два запуска без изменения аналитических правил и без чтения внутренней эталонной разметки в пользовательском API.

`ReviewPriority` хранит активный приоритет сигнала и структурированные факторы. `FinancialImpactSnapshot`, `OrganizationPrioritySnapshot` и `OrganizationComparisonSnapshot` сохраняют воспроизводимые срезы каждого запуска. Активные сигналы могут пересоздаваться анализом, а снимки организаций остаются доступными для реальной истории мини-графиков.

`RecurringPattern` является устойчивой сущностью с уникальным fingerprint. Текущие связи описывают `PatternSignal`, `PatternOrganization`, `PatternDoctor`, `PatternPatient` и `PatternService`. `SignalFingerprintHistory` и `PatternFingerprintHistory` сохраняют независимые снимки появлений. Старые таблицы review оставлены для совместимости схемы; новые маршруты используют единый журнал событий.

`ExpertFeedback` хранит структурированную оценку качества. `DecisionJournalIntegrityCheck` сохраняет результат проверки локальной hash-цепочки. Технические поля статуса денормализованы и синхронизируются с последним событием; источником экспертной истории является журнал.

`services/analytics.py` формирует два отдельных представления:

- командный центр — актуальные агрегаты и воспроизводимый приоритет;
- изменения — структурированное сравнение двух последних запусков.

Быстрый просмотр сигнала использует тот же сервис `services/signals.py`, что и полная карточка. Экспертное решение сохраняется существующим маршрутом, поэтому панель и карточка получают одинаковый обновлённый статус и историю.

## Воспроизводимость

Генератор, правила и Isolation Forest получают фиксированный seed. При одинаковой версии кода и входных данных формируются одинаковые записи, признаки, сигналы и оценки.

## Состояние интерфейса

Фильтры, страница и выбранный для быстрого просмотра сигнал хранятся в параметрах URL. Панель отображается поверх единого раздела «Проверка»; закрытие не пересоздаёт список и не меняет положение пользователя. Старый маршрут `/risk-signals` перенаправляется на `/signals`.

Полная карточка сигнала использует четыре якорных этапа: сводка, обоснование, связанные записи и решение специалиста. Риск, приоритет и финансовая значимость не пересчитываются: интерфейс только задаёт им разный визуальный вес. На странице существует один закреплённый компонент действий.

Раздел повторяющихся моделей использует URL для фильтров, сортировки и страницы. Карточка модели загружает сводку, временной ряд, граф и входящие сигналы отдельными типизированными запросами, но объединяет их в пять смысловых разделов. Граф возвращается как узлы и связи; интерфейс сначала показывает ключевые узлы, выделяет модель как центр и позволяет раскрыть дополнительные связи.

На ширинах меньше 1024 px списки сигналов и моделей используют карточное представление вместо широких таблиц. Подробная аналитика главной закрыта по умолчанию и не создаёт дополнительных API-запросов при раскрытии, поскольку данные уже получены существующим способом.

## Визуальный слой

Централизованные CSS-токены находятся в `frontend/app/globals.css` и экспортируются в Tailwind через `frontend/tailwind.config.ts`. Общие поверхности, кнопки, метрики, уведомления и закреплённые действия находятся в `frontend/components/ui.tsx`. Семантические индикаторы приоритета, риска, важности, устойчивости и статуса разделены по компонентам. Графики используют `frontend/lib/chart-theme.ts`.

Визуальный слой не меняет API и аналитические значения. Сценарий `frontend/scripts/capture-visuals.mjs` получает существующие данные и создаёт контрольные снимки вне репозитория.
