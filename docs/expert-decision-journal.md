# Журнал экспертных решений

Аналитический результат и экспертное решение хранятся раздельно. `RiskSignal` и `RecurringPattern` описывают обнаруженный аналитический контекст. `ExpertDecisionEvent` фиксирует действие специалиста.

## Append-only

Обычные API только добавляют события. Обновление и удаление событий не предоставляются. Уточнение решения создаёт новое событие с `supersedes_event_id`; исходное событие продолжает отображаться в хронологии.

Событие содержит тип и fingerprint объекта, текущий идентификатор при наличии, запуск анализа, действие, статус, структурированную причину, комментарий и технический профиль специалиста. Реальные персональные данные специалистов не используются.

Текущий статус рассчитывается по последнему событию цепочки. Поля статуса сигнала и модели остаются денормализованным представлением для быстрых выборок и обновляются в одной транзакции с новым событием. Журнал является главным источником экспертного статуса.

## Повторный анализ

После нового анализа fingerprint пересчитывается. Система находит прежнюю цепочку, восстанавливает экспертный статус и создаёт новый снимок появления без копирования событий. Если объект временно отсутствует, его журнал остаётся доступен. Полное пересоздание синтетического набора очищает журнал, поскольку меняет идентичность учебного набора; обычный повторный анализ историю сохраняет.

## API

- `GET /api/decision-journal`;
- `GET /api/decision-journal/{event_id}`;
- истории по идентификатору и fingerprint сигнала или модели;
- `POST .../decision-events`;
- совместимые `POST .../review`, переведённые на создание событий.
