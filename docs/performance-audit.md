# Аудит производительности публичной версии

## Цель и методика

Проверка выполнена 13 июля 2026 года для публичного развёртывания Vercel Services и
локального showcase-режима с snapshot `20260712`. Измерялись HTTP-задержка, размер JSON,
число SQL-запросов, время импорта приложения и время появления проверяемого рабочего
текста в браузере. Локальные значения не включают сетевую задержку Vercel. Значения
production после публикации фиксируются отдельным прогоном и могут меняться из-за региона
edge-узла и холодного старта.

Cold response `/api/health` на исходном production instance составил 10 887 мс.
Локальный импорт `app.main` занимал около 1 281 мс и загружал `pandas`, `numpy` и
`scikit-learn`. Копирование 23-МБ SQLite snapshot занимало около 13 мс и не являлось
основной причиной задержки.

SQL-профилирование исходной версии выявило 159 запросов для `pattern-summary`, 16 для
моделей организации, 23 для списка моделей и 12 для списка организаций. Основная причина
— получение связанных организаций и уникальных медицинских записей отдельными запросами.

## Выполненные оптимизации

- `GET /api/analytics/home` и `GET /api/analytics/overview` собирают подготовленные
  показатели в один версионированный ответ, не запуская анализ.
- Список сигналов больше не включает факторы, объяснение приоритета и fingerprint;
  подробности остаются в карточке и быстром просмотре.
- Списки фильтруются, сортируются и ограничиваются SQL до преобразования в Pydantic.
- Организации и участники моделей загружаются пакетно; сводка моделей считает уникальные
  записи одним SQL-запросом.
- Добавлены индексы для рабочих сортировок по запуску, приоритету, финансовой значимости,
  статусу, важности и устойчивости.
- Стабильные методические и аналитические GET-ответы получают общий HTTP cache policy;
  страницы с экспертным статусом не кэшируются на CDN.
- В браузере одинаковые GET объединяются, коротко переиспользуются и отменяются при
  смене фильтра. Любой POST инвалидирует клиентский и instance-local cache.
- Поиск в журнале отправляет запрос после паузы 350 мс.
- Граф связей, временные диаграммы, история и вторичные данные карточек загружаются при
  приближении раздела к viewport или раскрытии блока.
- Импорты генератора, анализа и ML-стека перенесены внутрь явных POST-команд. Обычный
  импорт serving-приложения не загружает `pandas`, `numpy` и `scikit-learn`.
- `pandas` и `scikit-learn` вынесены в optional extra `analysis`: Vercel serving bundle
  их не устанавливает, Docker и локальная разработка сохраняют полный конвейер.

## После оптимизации: Vercel production

Контрольный прогон выполнен через публичный адрес `https://verimed-gtc.vercel.app`
после публикации оптимизированной сборки. Первый прогон отражает прогрев edge и Python
instance, повторный — рабочий переход в уже посещённой публичной версии.

| Маршрут | Первый HTTP-ответ | Повторный HTTP-ответ | Полезное содержимое в прогретом браузере |
|---|---:|---:|---:|
| `/` | 1 165 мс | 612 мс | 1 156 мс |
| `/overview` | 1 228 мс | 583 мс | 1 222 мс |
| `/signals` | 928 мс | 663 мс | 662 мс |
| `/signals/840` | 990 мс | 709 мс | медиана 686 мс |
| `/organizations/18` | 875 мс | 791 мс | 769 мс |
| `/patterns/53` | 723 мс | 690 мс | 670 мс |
| `/api/health` | 1 254 мс | 671 мс | — |

Главная, список проверки и аналитический обзор укладываются в заданные ориентиры после
прогрева. Один первый браузерный прогон главной занял 4 092 мс из-за холодного запуска;
он не скрыт и не выдан за warm-результат. Повторные переходы оставались быстрее 1,3 с.
Ответы публичного API проходят через Vercel с сетевым нижним пределом около 0,6–0,8 с,
поэтому дальнейшее ускорение cold start зависит также от жизненного цикла serverless-instance.

## Ограничения измерений

- Browser Performance API недоступен в используемом контуре автоматизации, поэтому
  LCP/FCP не выдаются как точные лабораторные показатели. Зафиксированы HTTP TTFB и
  wall-clock до появления проверяемого рабочего текста.
- Первый cold start Vercel зависит от назначения региона и загрузки платформы.
- In-memory cache существует только в пределах живого Python instance.
- Экспертные POST-события не кэшируются; после записи обновлённые статусы читаются заново.

Значения production являются разовыми контрольными измерениями из региона исполнителя,
а не гарантированным SLA.
