# Оптимизация рабочего сценария

Дата: 13 июля 2026 года.

## Основной рабочий маршрут

Рабочая последовательность: «Сводная аналитика → Проверка → быстрый просмотр → карточка сигнала → решение → следующий сигнал». Переход к следующему объекту выполняется только по явному действию специалиста. После решения карточка сообщает результат и предлагает либо следующий сигнал из текущей выборки, либо возврат в «Проверку».

## Сохранение контекста

- фильтры, сортировка, страница, выбранное представление и открытый сигнал хранятся в URL;
- ссылка на полную карточку содержит `returnTo` и порядок идентификаторов текущей страницы `queueIds`;
- положение прокрутки перед выходом из быстрого просмотра сохраняется в `sessionStorage` отдельно для каждого URL очереди;
- при возврате очередь восстанавливает позицию один раз и удаляет использованное значение;
- предыдущий и следующий сигналы определяются только внутри текущей отфильтрованной страницы, поэтому переход не меняет рабочую выборку;
- список моделей использует тот же принцип для предыдущей и следующей модели.

## Быстрый просмотр

Первый экран панели показывает приоритет, статус, организацию, услугу, финансовую значимость и основную причину. Обоснование, связанные записи, ограничения и последнее решение закрыты по умолчанию. Для нового сигнала главное действие — «Начать проверку», для начатого — «Продолжить проверку». Полный набор экспертных решений остаётся в карточке сигнала.

Панель закрывается кнопкой, кликом по фону и клавишей Escape. После закрытия фокус возвращается к элементу, который открыл панель. Кнопки «Предыдущий сигнал» и «Следующий сигнал» учитывают текущую очередь.

## Иерархия действий

- в списке главное действие — открыть быстрый просмотр всей строкой или карточкой;
- в быстром просмотре видимо одно действие, продолжающее работу, и ссылка на полную карточку;
- в карточке сигнала основное решение — направление на углублённую проверку, остальные решения находятся в меню «Другие действия»;
- комментарий добавляется непосредственно из этапа решения без подтверждающего диалога;
- после сохранения появляется отдельное следующее действие, но автоматического перехода нет;
- одновременно не показывается более одного основного и двух вторичных действий.

## Факторы и связанные записи

В обосновании сначала отображаются три фактора с наибольшим приоритетом представления. Для каждого используется единая структура: вывод, фактическое значение, значение для сравнения, вклад и пояснение. Остальные факторы раскрываются по запросу. Связанные записи показаны временной последовательностью; первоначально видны четыре наиболее значимые записи.

## Мобильное поведение

- таблицы проверки, моделей и журнала решений заменяются карточками до ширины 1024 px;
- карточка очереди содержит только приоритет, статус, услугу, организацию, финансовую значимость и причину;
- журнал показывает карточки событий, а причина, комментарий и запуск открываются в боковой панели;
- этапы карточки сигнала остаются в закреплённой горизонтальной навигации с прокруткой внутри контейнера;
- граф связей на мобильном заменён списком ключевых связей, а граф открывается как дополнительное представление;
- корневой контейнер и карточки не расширяют документ шире viewport.

## Граф связей

Перед графом показан ответ на вопрос «Что формирует модель» и три основных вывода. По умолчанию доступны только ключевые связи. Выбор узла обновляет компактную карточку рядом с графом и не открывает новую страницу. Переход к объекту выполняется отдельной ссылкой. Центральный узел модели визуально выделен, дополнительные связи раскрываются явным действием.

## Доступность

- строки таблиц очереди и журнала доступны по Tab и открываются Enter или Space;
- интерактивные элементы внутри строки останавливают открытие строки;
- активный этап карточки имеет `aria-current="step"`;
- панели используют роль диалога, доступный заголовок, Escape и подписанную кнопку закрытия;
- выбранная строка обозначается не только цветом, но и `aria-pressed`;
- уведомления короткие и объявляются через `role="status"`;
- все кнопки без видимой подписи имеют `aria-label`.

## Границы этапа

Формулы, API аналитики, генератор данных, модели хранения и миграции не менялись. В API списка сигналов добавлен только опциональный фильтр `period_months`, необходимый для существующего фильтра периода. Новая визуальная айдентика, анимации, сохранённые представления и дополнительные аналитические показатели не добавлялись.

## Визуальная система

Последующий визуальный этап сохранил этот рабочий маршрут и добавил централизованные токены, отдельную семантику приоритета, риска, финансовой значимости, важности и устойчивости, светлые закреплённые таблицы, унифицированные мобильные карточки и спокойные микроинтеракции. Подробности находятся в `design-system.md` и `visual-language.md`.
