# Методика синтетических данных

Финансовые снимки используют существующие синтетические суммы медицинских записей. Новые исторические показатели создаются только при фактическом запуске анализа; искусственная история не генерируется.

## Состав

Набор полностью синтетический и не содержит реальных персональных данных:

- 20 вымышленных медицинских организаций;
- пять регионов Казахстана;
- 2 000 обезличенных пациентов;
- 100 обезличенных врачей;
- 30 медицинских услуг;
- период с 1 января по 30 июня 2026 года;
- 15 000 записей.

Seed по умолчанию — `20260712`.

## Нормальные записи

Сначала создаются 13 800 записей. Генератор учитывает допустимый профиль организации, врача этой организации, продолжительность услуги, занятость пациента, минимальный интервал, устойчивый диапазон стоимости и отсутствие точных дублей.

## Контролируемые внедрения

Затем добавляются по 150 записей каждого типа:

1. `exact_duplicate` — точный дубль;
2. `short_interval_repeat` — повтор раньше минимального интервала;
3. `temporal_conflict` — пересечение времени в разных организациях;
4. `excessive_frequency` — превышение частоты за 30 дней;
5. `price_deviation` — стоимость выше устойчивого диапазона;
6. `organization_profile_mismatch` — услуга вне профиля организации;
7. `end_of_month_spike` — концентрация в последние пять дней месяца;
8. `peer_group_anomaly` — отклонение структуры услуг организации.

Всего внедряется 1 200 аномальных записей — 8% набора. Каждая запись получает внутренние поля `is_ground_truth_anomaly`, `ground_truth_anomaly_type` и `anomaly_injection_id`. Они используются только для валидации и не входят в пользовательские API-контракты.

`GroundTruthAnomaly` сохраняет технический тип, связанные записи и параметры внедрения. Правила и модель не читают эталонную разметку.

## Данные для повторяющихся моделей

Модели строятся только по сформированным сигналам, связанным медицинским записям и сохранённым приоритетам. Эталонная разметка не участвует в группировке. Генератор и функции внедрения не изменялись для создания удобных моделей: группировка работает на существующем воспроизводимом наборе.

Для финансовой значимости каждой модели используется множество уникальных `MedicalRecord.id`. История модели появляется только после фактических запусков анализа; искусственные снимки не создаются.
