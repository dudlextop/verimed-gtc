# Финальный аудит готовности Verimed

Дата проверки: 13 июля 2026 года. Контрольный seed: `20260712`.

## 1. Общий статус

Код и документация Verimed готовы к передаче после устранения обнаруженных несогласованностей. Рабочий прототип запускается из чистой копии, миграции применяются к пустой базе, полный анализ воспроизводим, основные маршруты и адаптивные состояния работают.

До формальной подачи остаются внешние обязательные действия: создать первый Git-коммит и опубликовать репозиторий, подготовить презентацию на 7–10 слайдов, записать короткое видео и подтвердить новизну проекта относительно прошлых работ команды. Фактический Docker-запуск в среде аудита не выполнен из-за отсутствия Docker CLI.

## 2. Что проверено

- техническое задание GovTech Camp на шести страницах;
- README, продуктовые, аналитические, интерфейсные и эксплуатационные методики;
- генерация, анализ, финансовые агрегаты, повторяющиеся модели и экспертный журнал;
- 49 публичных API-маршрутов: 42 маршрута чтения и 7 рабочих POST-сценариев;
- 15 маршрутов frontend, включая перенаправления `/risk-signals`, `/data` и `/settings`;
- 12 ключевых экранов на ширинах 1440, 1280, 1024, 768 и 375 px — 60 адаптивных проверок без горизонтального переполнения документа;
- быстрый просмотр, карточки, журнал, граф связей, прямые URL, загрузка, ошибка и пустые состояния;
- клавиатурный фокус, Escape, текстовые названия действий, `prefers-reduced-motion` и консоль браузера;
- 27 воспроизводимых визуальных снимков;
- четырёхстраничное альбомное печатное представление `/overview` без интерактивных действий;
- запрещённая терминология, абсолютные пользовательские пути, секреты и чувствительные данные.

Автоматические проверки: 83 backend-теста и 60 frontend-тестов пройдены; Ruff, mypy strict, ESLint, TypeScript strict и production build завершены без ошибок. Единственное backend-предупреждение относится к будущей замене тестовой интеграции Starlette/httpx.

## 3. Исправленные ошибки

1. Аналитический обзор показывал 1 442 записи из-за обратного умножения округлённой доли `0,0961` на 15 000. Теперь точный счётчик равен `TP + FP = 1 441`.
2. В сводке были жёстко заданные проценты изменений. Теперь изменения рассчитываются по двум сохранённым запускам; при отсутствии базы сравнения выводится недоступное состояние.
3. Порог организаций повышенного риска отличался между расчётом и пояснением. Он согласован со значением 60.
4. Сводная финансовая сумма могла расходиться с каноническим снимком уникальных записей. Сводка и изменения теперь используют `FinancialImpactSnapshot`.
5. Десятичные значения факторов местами выводились с точкой и без форматирования. Пользовательские объяснения используют русский формат и разделители разрядов.
6. `/data` и `/settings` выглядели как незавершённые функции при прямом открытии. Они перенаправляют на рабочую главную страницу.
7. Мобильный быстрый просмотр обрезал длинное название и бейдж приоритета. Добавлены безопасный перенос и компактное расположение.
8. `npm start` использовал несовместимую с `output: standalone` команду. Скрипт запускает фактический standalone-сервер и подготавливает статические файлы.
9. Docker-стек не применял миграции при старте и не ожидал готовности backend. Добавлены Alembic-команда, healthcheck и `service_healthy`; контексты очищены `.dockerignore`.

## 5. Состояние репозитория

- `.gitignore` исключает `.env`, базы, `.venv`, `node_modules`, `.next`, кэши и coverage;
- локальные окружения, сборка и SQLite-файлы присутствуют в рабочем каталоге, но игнорируются;
- абсолютные пути вида `/Users/...` в коде и пользовательской документации не найдены;
- секреты, ключи и приватные сертификаты не найдены;
- `.env.example` содержит только локальные адреса и параметры базы;
- бинарные визуальные снимки сохраняются во временный каталог;
- имена исходных файлов совместимы с Linux;
- лицензия не добавлена: техническое задание её не требует, но перед публичной публикацией команда должна выбрать режим лицензирования;
- текущий локальный Git не содержит коммитов и удалённого репозитория.

## 6. Чистый локальный запуск

Создана изолированная копия во временном каталоге без `.git`, `.venv`, `node_modules`, `.next`, SQLite, кэшей и сгенерированных артефактов.

В чистой копии:

- создано новое окружение Python 3.12.13;
- выполнено `pip install -e '.[dev]'`;
- применены миграции `0001`–`0006` к пустой SQLite-базе;
- `python -m app.seed` создал 15 000 записей, 1 441 сигнал и 73 модели;
- `npm ci` установил зависимости только из lock-файла;
- production build завершён успешно;
- backend и frontend отвечали на локальных адресах.

В системе отсутствовала команда `python3.12`, поэтому использован предоставленный средой интерпретатор Python 3.12.13; остальные шаги соответствовали README. Node.js: 20.19.5, npm: 10.8.2.

## 7. Docker

Docker CLI отсутствует (`command not found`), поэтому фактическая сборка контейнеров не выполнялась.

Статически проверены:

- три сервиса `db`, `backend`, `frontend`;
- корректные build context;
- PostgreSQL healthcheck;
- применение Alembic перед Uvicorn;
- backend healthcheck `/api/health`;
- ожидание `service_healthy` между сервисами;
- build-time адрес API frontend;
- standalone-команда Next.js;
- `.dockerignore` frontend и backend.

## 8. Конфиденциальность

Набор полностью синтетический. В исходных данных и интерфейсе отсутствуют ФИО пациентов, ИИН, адреса, телефоны, реальные медицинские истории и реальные учётные данные. Пациенты и врачи представлены кодами. Профиль специалиста явно вымышленный. Ошибка backend верхнего уровня возвращает нейтральный русский текст без stack trace. ## 11. Известные ограничения

- качество измерено только на синтетической разметке;
- нет продукционной авторизации, фоновой очереди и мониторинга;
- экспертная обратная связь не переобучает модель;
- малые сопоставимые группы снижают надёжность сравнения;
- hash-цепочка обнаруживает локальные изменения, но не обеспечивает внешнюю юридическую гарантию;
- `Recharts` 2 сообщает о режиме сопровождения, а `Vitest` — о deprecated CJS API; обновление зависимостей не выполнялось в финальном аудите;
- тестовая зависимость Starlette выводит одно предупреждение о будущем переходе на `httpx2`.

## 12. Блокирующие проблемы

Для работы кода блокирующих проблем не найдено.

Для формальной сдачи блокируют пакет материалов:

1. нет первого Git-коммита и опубликованного GitHub-репозитория;
2. не подготовлена презентация на 7–10 слайдов;
3. не записано короткое видео;
4. не подтверждена командой новизна относительно Decentrathon 5.0.

## 13. Неблокирующие проблемы

- Docker необходимо один раз фактически собрать и запустить на машине с Docker CLI;
- перед публичной публикацией следует определить лицензию;
- отдельные карточки остаются информационно плотными, но основной маршрут не блокируют;
- печатный обзор занимает четыре альбомные страницы; это сохраняет читаемость и ключевые блоки без разрывов.

## 14. Готовность к сдаче
