# Verimed

Verimed — интеллектуальная система прозрачности и контроля медицинских услуг. Она анализирует обезличенные сведения, формирует объяснимые сигналы риска и помогает специалисту определить очерёдность экспертной проверки. Окончательный вывод всегда принимает уполномоченный специалист.

## Проблема

Большой поток медицинских услуг сложно просматривать вручную целиком. Точные дубли, нетипичная частота, временные пересечения, отклонения стоимости и устойчивые повторяющиеся модели распределены по организациям и периодам. Специалисту нужен не ещё один реестр, а воспроизводимый способ сократить выборку, увидеть причины приоритета и сохранить экспертное решение.

## Основной пользователь

Специалист по контролю медицинских услуг, который:

1. оценивает текущий объём приоритетной работы;
2. изучает сигнал или повторяющуюся модель;
3. проверяет связанные записи и сравнение с сопоставимой группой;
4. фиксирует решение и комментарий;
5. переходит к следующему объекту, сохраняя контекст очереди.

## Что делает Verimed

- генерирует воспроизводимый синтетический набор и скрытую эталонную разметку;
- выполняет семь детерминированных проверок;
- оценивает поведение организаций через Isolation Forest внутри сопоставимых групп;
- рассчитывает объяснимую оценку риска, приоритет проверки и финансовую значимость;
- группирует связанные сигналы в восемь типов повторяющихся моделей;
- показывает факторы, значения для сравнения и ограничения анализа;
- сохраняет экспертные действия в append-only журнале и восстанавливает историю по fingerprint;
- измеряет качество обнаружения на известной синтетической разметке.

## Что система не делает

- не использует реальные медицинские истории;
- не принимает окончательное решение;
- не блокирует оплату автоматически;
- не выполняет клиническую интерпретацию;
- не обучается автоматически на экспертной обратной связи;
- не предоставляет продукционную авторизацию и юридически значимый аудит.

## Архитектура

```text
Синтетические справочники и нормальные записи
  → контролируемые внедрения и эталонная разметка
  → детерминированные проверки
  → сопоставимые группы и признаки организаций
  → Isolation Forest
  → риск, приоритет и финансовая значимость
  → повторяющиеся модели и экспертный журнал
  → FastAPI + PostgreSQL/SQLite
  → Next.js
```

- frontend: Next.js App Router, TypeScript strict, Tailwind CSS, Recharts, TanStack Table;
- backend: Python 3.12, FastAPI, SQLAlchemy, Pydantic, pandas, scikit-learn;
- хранение: PostgreSQL через Docker Compose, SQLite для локальной проверки и тестов;
- миграции: Alembic;
- бизнес-логика: `backend/app/services`;
- генерация: `backend/app/synthetic`.

Подробности: [docs/architecture.md](docs/architecture.md).

## Данные

Набор полностью синтетический:

- 20 вымышленных медицинских организаций;
- пять регионов Казахстана;
- 2 000 обезличенных пациентов;
- 100 обезличенных врачей;
- 30 медицинских услуг;
- 15 000 записей за шесть месяцев;
- 1 200 контролируемо внедрённых аномальных записей.

Сначала создаются 13 800 нормальных записей, затем восемь независимых функций внедряют по 150 аномальных записей каждого типа. Правила и модель не читают эталонную разметку. Методика описана в [docs/data-methodology.md](docs/data-methodology.md).

## AI/Data-driven компонент

Verimed сочетает правила и поиск аномального поведения:

- правила обнаруживают интерпретируемые отклонения на уровне записей;
- сопоставимые группы ограничивают сравнение регионом, типом, специализацией и масштабом организации;
- Isolation Forest оценивает многомерное отклонение поведения организации;
- детерминированная группировка выделяет устойчивые повторяющиеся модели;
- скоринг и ранжирование формируют рабочий приоритет.

Языковая модель не используется для обнаружения, расчёта или объяснения.

## Объяснимость

Карточка сигнала показывает:

- что обнаружено;
- фактическое и типичное значения;
- вклад каждого фактора;
- связанные записи;
- ограничения доступных данных;
- отдельные оценку риска и приоритет проверки.

Экспертное решение хранится отдельно от аналитического результата. Подробнее: [docs/risk-methodology.md](docs/risk-methodology.md), [docs/review-priority-methodology.md](docs/review-priority-methodology.md), [docs/expert-decision-journal.md](docs/expert-decision-journal.md).

## Ограничения

- показатели качества не перенесены на реальные медицинские данные;
- малые сопоставимые группы требуют расширения критериев;
- доступные сведения не содержат полной клинической истории;
- одинаковая последовательность услуг может иметь медицинское объяснение;
- финансовая значимость отражает масштаб связанных услуг;
- hash-цепочка журнала обнаруживает локальные изменения, но не является внешней криптографической гарантией.

## Быстрый запуск через Docker Compose

Требуются Docker Engine 24+ и Docker Compose v2.

```bash
cp .env.example .env
docker compose up --build
```

Backend применит Alembic-миграции, при пустой базе создаст синтетический набор и выполнит анализ. После запуска:

- интерфейс: <http://localhost:3000>
- аналитический обзор: <http://localhost:3000/overview>
- документация API: <http://localhost:8000/docs>
- состояние API: <http://localhost:8000/api/health>

Остановка:

```bash
docker compose down
```

Полный сброс локального тома:

```bash
docker compose down -v
```

## Локальный запуск без Docker

Backend требует Python 3.12:

```bash
cd backend
python3.12 -m venv .venv
source .venv/bin/activate
python -m pip install -e '.[dev]'
alembic upgrade head
python -m app.seed
uvicorn app.main:app --reload
```

Frontend требует Node.js 20+; во втором терминале:

```bash
cd frontend
npm ci
NEXT_PUBLIC_API_URL=http://localhost:8000/api npm run dev
```

## Развёртывание через Vercel Services

Корневой `vercel.json` описывает два сервиса: Next.js из `frontend/` и FastAPI из `backend/`. Публичные запросы `/api/*` направляются в backend, остальные маршруты — во frontend. Поэтому в Vercel `NEXT_PUBLIC_API_URL` можно не задавать или установить равным `/api`.

В настройках проекта Vercel необходимы:

```text
DATABASE_URL=postgresql://<пользователь>:<пароль>@<хост>/<база>
NEXT_PUBLIC_API_URL=/api
AUTO_BOOTSTRAP_DATABASE=false
```

`DATABASE_URL` должен указывать на внешнюю постоянную PostgreSQL-базу. Локальная SQLite при запуске в Vercel отклоняется конфигурацией. Стандартные URL `postgres://` и `postgresql://` автоматически используют установленный драйвер psycopg.

Миграции и исходное заполнение выполняются один раз до первого рабочего запуска, а не при обработке запросов:

```bash
cd backend
DATABASE_URL='<production-url>' AUTO_BOOTSTRAP_DATABASE=false alembic upgrade head
DATABASE_URL='<production-url>' AUTO_BOOTSTRAP_DATABASE=false python -m app.seed
```

После этого приложение запускается без `create_all` и генерации данных в lifespan Vercel-сервиса. Контрольный endpoint доступен по `/api/health`. Для использования Services проекту должен быть предоставлен доступ к этой возможности Vercel.

## Основные маршруты интерфейса

- `/` — сводная аналитика и командный центр;
- `/overview` — управленческий аналитический обзор;
- `/signals` — единый раздел проверки;
- `/signals/{id}` — карточка сигнала;
- `/organizations` и `/organizations/{id}` — медицинские организации;
- `/patterns` и `/patterns/{id}` — повторяющиеся модели;
- `/reviews` — результаты экспертной оценки;
- `/decision-journal` — журнал решений;
- `/methodology` — методика анализа.

Старый маршрут `/risk-signals` и технические `/data`, `/settings` перенаправляются на рабочие страницы.

## API

Основные группы:

- `/api/analytics/*` — сводка, изменения, финансовая значимость, приоритеты и модели;
- `/api/analysis/*` — метрики и запуск анализа;
- `/api/signals/*` — сигналы, быстрый просмотр, история и решения;
- `/api/organizations/*` — организации, сравнение и финансовые снимки;
- `/api/patterns/*` — модели, граф, временной ряд и экспертная оценка;
- `/api/decision-journal/*` — журнал и проверка целостности;
- `/api/methodology` — пользовательская методика.

Полный интерактивный перечень доступен по `/docs` после запуска backend.

## Конфиденциальность и данные

- в проекте используются только синтетические сведения;
- пациенты и врачи представлены обезличенными кодами;
- набор не содержит ФИО пациентов, ИИН, адресов, телефонов или реальных медицинских историй;
- профиль специалиста вымышленный и не является учётной записью;
- `.env` исключён через `.gitignore`, а `.env.example` содержит только локальные значения;
- пользовательские ошибки API не содержат stack trace;
- локальные базы, окружения, кэши и визуальные снимки исключены из состава репозитория.
